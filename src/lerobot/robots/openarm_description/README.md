# OpenArm 2.0 robot description (kinematics-only)

Per-arm URDFs for Pink/pinocchio Cartesian IK, used by
`bi_openarm_follower` to drive the two OpenArm 2.0 arms from a Cartesian
teleop (Quest VR).

## Provenance

Derived from the official [enactic/openarm_description](https://github.com/enactic/openarm_description)
v2.0 bimanual URDF (`output.urdf`, autogenerated from
`assets/robot/openarm_v2.0/urdf/openarm_v20.urdf.xacro`). Apache-2.0 —
see `LICENSE.txt` in this directory.

Extraction (mechanical, reproducible with `xml.etree`):

- split the bimanual model into one URDF per arm, with
  `openarm_{side}_base_link` as the root;
- kept the `openarm_{side}_joint1..7` revolute chain and the
  `openarm_{side}_ee_base_link` wrist/gripper-flange frame (the IK target);
- dropped the torso links, mount joints, and finger links/joints (the
  gripper motor is not IK-tracked);
- stripped all `<visual>`/`<collision>` geometry. Pinocchio refuses to
  parse a URDF whose referenced mesh files are missing, and FK/IK need no
  geometry — this keeps the vendored files small (no multi-MB meshes).

Joint origins, axes, and limits are verbatim from upstream, so the
kinematics are identical to the upstream model.

## Conventions (verified against the URDF and the dora OpenArm stack)

- **Joint zero = motor zero**: all-zero joint angles put the arm hanging
  straight down, matching the OpenArm 2.0 motor factory zero and the
  `openarm_bimanual.xml` MuJoCo model's `qpos=0`. FK at q=0 places
  `ee_base_link` at `(0, ±0.1225, -0.436)` m (left/right) with identity
  orientation — the same EE zero the validated dora VR stack hardcodes
  (`_EE_ZERO`). No `(sign, offset)` motor->URDF alignment is needed
  (contrast with `so107_description/joint_alignment.py`).
- **Joint ordering**: `joint_1..joint_7` motor order maps 1:1 onto
  `openarm_{side}_joint1..7`.
- **Base frame**: +X forward (the direction the arms reach), +Y left,
  +Z up. The left arm base sits at +Y, the right at -Y. The gripper
  flange (`ee_base_link`) is the IK target frame — same control point as
  the `*_ee_control_point` sites in the upstream MuJoCo model.
